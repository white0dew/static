# Umami 接入设计

## 背景

当前项目是纯静态站点，入口文件为 `index.html`，通过虚拟主机或 FTP 方式部署。站点没有前端构建流程，也没有单页路由。

用户已提供可用的 Umami 脚本地址和 `website-id`，目标是在尽量小改动的前提下，为首页增加访问统计能力。

## 目标

- 在当前首页接入 Umami 统计
- 不改动现有页面结构和交互逻辑
- 不引入打包、环境变量或额外依赖
- 保持后续排查和维护成本最低

## 非目标

- 不为不存在的其他 HTML 页面补统计脚本
- 不实现按环境切换统计配置
- 不新增自定义事件埋点
- 不调整现有 CSS、动画或业务文案

## 方案对比

### 方案 A：直接在 `index.html` 的 `<head>` 中接入官方脚本

做法是在首页静态写入 Umami 提供的脚本标签。

优点：

- 改动最小，最符合当前项目形态
- 与 Umami 官方接入方式一致，排障简单
- 不影响现有 `js/main.js` 的职责边界

缺点：

- 如果未来新增多个独立 HTML 页面，需要逐页补充

### 方案 B：在 `js/main.js` 中动态插入脚本

优点：

- 理论上便于集中管理

缺点：

- 对当前纯静态站是过度设计
- 统计初始化链路更长，问题更隐蔽

### 方案 C：引入部署替换或多环境配置

优点：

- 适合多环境部署

缺点：

- 当前项目没有构建流程，维护成本高于收益

## 推荐方案

采用方案 A，直接在 `index.html` 的 `<head>` 中插入 Umami 官方脚本。

原因：

1. 与现有纯静态部署方式最匹配
2. 不需要修改本地 JavaScript 逻辑
3. 上线后出现统计异常时，定位路径最短

## 设计细节

### 修改点

仅修改 `index.html`：

- 在现有本地脚本标签附近插入 Umami 统计脚本
- 保持现有资源版本号策略不变

### 接入脚本

```html
<script
  defer
  src="https://umami.aistar.cool/script.js"
  data-website-id="8f4c108a-d47c-4449-aa25-66abdbd8303e"
></script>
```

### 放置位置

放在 `index.html` 的 `<head>` 中，和当前样式、站点脚本一起加载。

### 数据范围

本次仅覆盖当前首页 `index.html`。如果后续新增文章页、落地页或其他独立 HTML 文件，需要重复添加同一脚本，才能纳入统计。

## 风险与处理

### 风险 1：统计脚本被网络策略或浏览器插件拦截

处理方式：

- 页面功能不依赖 Umami，统计失败不会影响站点展示
- 上线后通过浏览器开发者工具确认脚本请求是否成功

### 风险 2：未来页面增加后遗漏统计脚本

处理方式：

- 将“新增 HTML 页面时同步补统计脚本”纳入发布检查项

## 验证方案

实施后进行以下验证：

1. 检查 `index.html` 中脚本标签位置和属性是否正确
2. 本地打开页面确认无明显报错
3. 部署后在浏览器网络面板确认 `script.js` 成功加载
4. 在 Umami 后台确认页面访问数据开始出现

## 实施范围总结

这是一次单文件、小范围、低风险的静态接入：

- 改 1 个文件：`index.html`
- 不改 `css/style.css`
- 不改 `js/main.js`
- 不引入兼容代码
